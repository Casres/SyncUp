## HARD RULES — Round 17 additions (Messaging · DM / Group / Event chat)
================================================================
R17-1. MESSAGES IS THE THIRD SEGMENT OF THE FRIENDS TAB. The inbox
       lives inside FriendsListScreen as the third option in the
       existing SegmentedSwitcher (SCREENS.md → Friends List). The
       switcher is now 3-way: Friends · Groups · Messages, in that
       fixed left-to-right order. It is a carousel that wraps in BOTH
       directions:
         · Messages · swipe right → Friends.
         · Friends · swipe left → Messages.
       Selecting Messages renders the inbox list in place of the
         friend-row body; the FlowHeader and floating "+" / Scan FAB
         behavior are unchanged for the Friends/Groups segments.
       The inbox is NOT a tab, NOT a route, NOT a standalone screen —
         it is a segment view inside FriendsList (route: `FriendsList`).
       NEVER add a fifth bottom-tab for messages. NEVER reorder the
         SegmentedSwitcher (Friends · Groups · Messages is locked).
         NEVER break the both-directions carousel wrap.

R17-2. INBOX ROW ANATOMY IS FIXED. Each conversation row in the
       Messages segment renders, left-to-right:
         · Avatar (circular · type-specific per R17-3) · leading.
         · Conversation name · bodyMed ink · top line · 1-line clamp
           (R5-6).
         · Last-message preview · caption ink2 · bottom line · 1-line
           truncated (R5-6).
         · Timestamp · caption ink3 · top-right.
         · Unread badge · trailing, when the conversation has unread
           messages.
       Rows are sorted by last-message recency (lastMessageAt, newest
         first; see R17-13 data model). Archived conversations are
         excluded from the inbox entirely (R17-7).
       Tap row → opens the thread for that conversation per the
         routing in R17-11. Row-tap haptic is `light` (consistent
         with FriendsList row tap · SCREENS.md).
       NEVER render a non-circular avatar in the inbox. NEVER show
         more than one line of preview text.

R17-3. INBOX AND HEADER AVATARS ARE CIRCULAR AND TYPE-SPECIFIC. All
       three conversation types render a circular avatar; the content
       differs by type and is identical between the inbox row (R17-2)
       and the thread header (R17-4):
         · 1:1 DM → the friend's circular profile photo.
         · Group chat → a CLUSTER of up to 5 overlapping circular
           member avatars. The icon count matches the real member
           count up to a cap of 5 (2 members → 2 avatars; 6+ members →
           5 avatars). The cluster does NOT render a "+N" overflow
           count — it caps silently at 5.
         · Event chat → the event photo, circular crop.
       NEVER render a square or rounded-rect avatar in messaging
         surfaces. NEVER show a participant count anywhere on the
         cluster (counts live on the group/event page per R17-4).

R17-4. THREAD HEADER IS PER-TYPE AND OMITS PARTICIPANT COUNT. Each
       thread is a routed screen whose header renders, by type:
         · 1:1 DM → friend's name + circular profile photo.
         · Group chat → group name + cluster avatar (same cluster as
           the inbox row · R17-3).
         · Event chat → event name + event photo (circular crop).
       The header NEVER shows a participant/member count — that
         information lives on the event page (EventDetail) or the
         group page (GroupDetail), not in the thread. The header
         carries a back affordance returning to the caller per the
         nav graph (R17-12).
       NEVER surface participant counts, member lists, or roster
         management in the thread header in this round.

R17-5. MESSAGE BUBBLES ARE SIDE-AND-COLOR CODED; SENDER LABELS ARE
       GROUP/EVENT ONLY. Within a thread's message list:
         · SENT messages (authored by the current user) → right-
           aligned · accent color fill.
         · RECEIVED messages → left-aligned · bgElevated fill.
       Sender label rule:
         · Group chat + event chat → the sender's NAME appears ABOVE
           the bubble, left-aligned, next to their avatar, on
           received messages.
         · 1:1 DM → the sender label is OMITTED entirely (the header
           already names the single counterpart).
       Use design tokens for both fills (accent, bgElevated) — never
         hardcode hex (Hard Rule · src/theme/colors.ts).
       NEVER show a sender label on a 1:1 thread. NEVER show a sender
         label on the current user's own sent bubbles.

R17-6. TIMESTAMPS ARE GAP-GATED AND DATE-AWARE. A timestamp renders
       in the message list:
         · On the FIRST message of a conversation, and
         · After any gap of 5 OR MORE minutes between consecutive
           messages.
       Format:
         · Today → clock time only: "3:42 PM".
         · Older than today → date + time: "Jun 1 · 3:42 PM".
       Timestamps that do not meet the gap rule are NOT rendered —
         consecutive messages within 5 minutes share the most recent
         visible timestamp above them.
       NEVER render a per-message timestamp on every bubble. NEVER use
         the older-format ("Jun 1 · …") for messages sent today.

R17-7. TYPING INDICATOR IS A RECEIVED-STYLE BUBBLE. While a remote
       participant is typing (driven by `chat:typing:start` /
       `chat:typing:stop` · R17-13), the thread renders three animated
       dots styled as a RECEIVED bubble at the bottom of the message
       list:
         · Left-aligned · bgElevated fill (same as a received bubble
           per R17-5).
         · Sender label follows the received-bubble rule exactly —
           name + avatar above the dots for group/event chats, omitted
           for 1:1.
       The indicator is loading-state-adjacent but is NOT a spinner —
         it is the canonical three-dot typing animation and is the one
         permitted exception to "spinner only" for this affordance.
       NEVER show read receipts alongside the typing indicator in V1
         (R17-14). NEVER render the typing bubble for the current
         user's own typing.

R17-8. INPUT BAR IS TEXT-ONLY WITH A HIDE-WHEN-EMPTY SEND ICON. The
       thread input bar renders a single-line-growing text field plus
       a paper-plane / arrow SEND icon:
         · The send icon is HIDDEN COMPLETELY when the input is empty;
           it appears only once the field contains text.
         · Messages are text only. Native emoji via the device
           keyboard are allowed. There is NO media attachment, NO file
           picker, NO reactions affordance (R17-15).
       NEVER render a disabled/greyed send icon on an empty field —
         it is hidden, not disabled. NEVER add a media, camera, file,
         or reaction button to the input bar in this round.

R17-9. 1:1 DM CONVERSATIONS ARE CREATED ON FIRST MESSAGE SENT. A DM
       conversation does NOT exist until the first message is actually
       sent. Tapping a DM entry point opens the thread surface, but no
       Conversation row is persisted (and none appears in the inbox)
       until that first message lands. Entry points are exactly two:
         · Friend Profile "DM" button (R16-5) — its R16-9 stub is now
           SUPERSEDED by this round: the button opens a real DM thread
           instead of firing the "coming soon" toast.
         · Long-press a friend row in FriendsListScreen → context menu
           with EXACTLY two options, in this order: "Message" +
           "View Profile". "Message" opens the DM thread; "View
           Profile" pushes FriendProfile (R16-2).
       Get-or-create resolution routes through
         `POST /conversations/direct/:friendId` (R17-13) so a second
         entry to an existing DM reuses the same conversation.
       NEVER create an empty DM conversation on DM-button tap or on
         long-press "Message" tap. NEVER add a third option to the
         friend-row long-press menu in this round.

R17-10. GROUP CHATS ARE AUTO-CREATED WITH THE GROUP; EVENT CHATS ARE
       HOST-ENABLED. The two non-DM conversation types differ in
       lifecycle:
         · GROUP CHAT → auto-created at the moment a FriendGroup is
           created (`POST /groups/:groupId/chat`, auto-called on group
           creation · R17-13). ALL group members are participants from
           the start. When a new member joins the group later, they
           are added to the chat automatically.
         · EVENT CHAT → does NOT exist until the host manually enables
           it from the event surface (`POST /events/:id/chat` ·
           R17-13). Once enabled, participants = ALL invitees.
       Event chat lifecycle:
         · ONE-TIME event → the chat ARCHIVES 48 hours after the
           event's end time (the "thank you / recap" window). Archived
           conversations are excluded from the inbox (R17-2).
         · RECURRING event → NEVER archives; stays active indefinitely.
       NEVER auto-enable an event chat — it is always an explicit host
         action. NEVER archive a recurring-event chat. NEVER archive a
         group or DM conversation in this round (archiving is EVENT-only
         · R17-13 data model).

R17-11. NOTIFICATION CARDS FOR MESSAGES ROUTE STRAIGHT TO THE THREAD.
       A new-message notification renders as a card in the NotifSheet
       with type-specific anatomy:
         · Avatar · type-specific per the inbox rules (R17-3).
         · Conversation name.
         · Message preview · 1-line.
         · Timestamp.
       Tap behavior follows the navigating-card convention (per R12-1):
       the tapped card fires a `medium` haptic and routes DIRECTLY to
       the thread while the NotifSheet executes its dismiss transition
       CONCURRENT with the thread mounting (the sheet does not remain
       open behind the thread; back from the thread returns to the
       activated tab's top-of-stack, not to the sheet). Per-type
       routing:
         · DM card → Friends tab activates → Messages segment → DM
           thread.
         · Group chat card → Friends tab activates → Messages segment →
           group thread.
         · Event chat card → Home tab activates → event thread.
       NEVER open the inbox segment as an intermediate stop before the
         thread — the tap lands on the thread directly. NEVER leave the
         NotifSheet open behind the thread (R12-1).

R17-12. NAV GRAPH — MESSAGING THREADS LIVE IN TWO STACKS. Threads are
       routed screens, placed to match each conversation type's home
       surface and consistent with NAVIGATION.md:
         · DM threads and GROUP threads live in the FriendsStack,
           reached from the Messages segment of FriendsList (R17-1) and
           from message notification cards (R17-11). Suggested route:
           `MessageThread` with params
           { conversationId: string; type: 'DIRECT' | 'GROUP' }.
         · EVENT threads live in the HomeStack, reached from the event
           surface (EventDetail) and from event-chat notification cards
           (R17-11). Suggested route: `EventChat` with params
           { conversationId: string; eventId: string }.
       Transition is a step push (240ms easeStd) within the owning
         stack; cross-stack arrivals from a notification card switch
         the tab first, then push (per R12-1 / NAVIGATION.md cross-
         stack pattern). Back from a thread returns to that stack's
         prior top-of-stack (the Messages segment for DM/group; the
         event surface for event chat) — NEVER back to the NotifSheet.
       Exact route names and param shapes are NON-NORMATIVE here and
         finalize in R18; the placement (FriendsStack for DM/group,
         HomeStack for event) IS normative.
       NEVER place an event thread in the FriendsStack. NEVER place a
         DM or group thread in the HomeStack.

R17-13. DATA MODEL AND TRANSPORT (NON-NORMATIVE · GUIDES R18 BUILD).
       The following is NOT a UI hard rule. It records the conversation
       schema, real-time transport, and REST surface so R18 agents
       build against one shape. Treat names/shapes as a guide, not a
       locked contract — finalize during R18.

       Conversation
         id
         type: DIRECT | GROUP | EVENT
         linkedGroupId?      (GROUP only)
         linkedEventId?      (EVENT only)
         createdAt
         lastMessageAt       (inbox sort key · R17-2)
         archivedAt?         (EVENT only — set 48h after event.endsAt
                              for one-time events · R17-10)

       Message
         id
         conversationId
         senderId
         content: string
         sentAt

       ConversationParticipant
         conversationId
         userId
         joinedAt
         lastReadMessageId?  (nullable — reserved for future read
                              receipts · R17-14)

       Socket.io events:
         · chat:message:new
         · chat:typing:start
         · chat:typing:stop
         · chat:conversation:new   (notify a participant when they are
                                    added to a new conversation)

       REST endpoints:
         · GET  /conversations               inbox (sorted by
                                              lastMessageAt; excludes
                                              archived)
         · GET  /conversations/:id/messages  paginated thread
         · POST /conversations/direct/:friendId
                                              get-or-create 1:1 (R17-9)
         · POST /conversations/:id/messages  send a message
         · POST /events/:id/chat             host enables event chat
                                              (R17-10)
         · POST /groups/:groupId/chat        auto-called on group
                                              creation (R17-10)

R17-14. READ RECEIPTS ARE EXPLICITLY DEFERRED TO V2. V1 messaging ships
       with NO read-receipt UI. The schema reserves a hook for it
       (`ConversationParticipant.lastReadMessageId`, nullable · R17-13)
       but nothing renders a read/seen state in the inbox, thread, or
       notification card in this round.
       NEVER render a "Seen", "Read", or delivery-state indicator on
         any messaging surface in V1.

R17-15. R17 / V1 MESSAGING SCOPE BOUNDARIES. The following are
       explicitly OUT OF SCOPE for this round and must not be designed,
       stubbed, or built until a future round specs them:
         · Read receipts (V2 · R17-14).
         · Media messages, file sharing, and message reactions (input
           bar is text + native emoji only · R17-8).
         · Message editing and message deletion.
         · Group chat admin controls beyond auto-creation (no add/
           remove participant, no rename-from-thread, no leave flow in
           this round · R17-10).
         · Event chat host moderation controls (no mute, kick, delete-
           message, or close-early in V1 — enabling the chat is the
           only host control · R17-10).
       NEVER introduce any of the above affordances on a messaging
         surface in R17/V1. When one is promoted, it supersedes the
         relevant clause here in a future round.

## HARD RULES — Round 16 additions (Friend Profile · QuickProfileSheet drill)
================================================================
R16-1. FRIEND PROFILE IS A FULL SCREEN, NOT A SHEET. The Friend Profile
       surface is a routed screen inside FriendsStack
       (route: `FriendProfile`, params: { friendId: string }). It is
       NEVER presented as a sheet, modal, or transient overlay. The
       anatomy is fixed (SCREENS.md → Friend Profile):
         1. FlowHeader · back · title "Profile" · trailing overflow
            (⋯) icon (R16-6).
         2. Profile hero card · 72px RingAvatar paired with AvailDot +
            text per R5-1 · h2 name · mono ink3 @handle · CategoryBadge.
         3. StatTile row (4-up) · HOSTED · ATTENDED · FRIENDS · GROUPS.
         4. FRIEND TYPES section · chips · display-only per R16-10.
         5. MUTUAL EVENTS section · read-only list per R16-11.
         6. Action row · Make Plans + DM (two equal primary pills) per
            R16-5. Remove friend, Block, and Report all live in the
            header overflow (R16-6) — NOT in this row.
       Friend Profile is the ONLY surface that may display friends-only
       fields (availability ring, friend count, group count, friend
       type assignments). QuickProfileSheet (R12-6, R15-2) remains the
       public limited-data surface and is unaffected by this round.
       NEVER render Friend Profile for a non-friend. Entry resolution
         is the caller's responsibility (R16-2).

R16-2. FRIEND PROFILE ENTRY POINTS ARE EXPLICITLY ENUMERATED. The only
       routes that may push `FriendProfile` are:
         (a) FriendsListScreen · row tap on a confirmed friend.
         (b) FriendTypesManager · type-member row tap on a confirmed
             friend (existing behavior · SCREENS.md).
         (c) GroupDetailScreen · Members row tap on a confirmed friend
             (SCREENS.md → "Members row → push FriendProfile (if
             non-self)").
       The following surfaces explicitly do NOT push Friend Profile:
         · AttendeesSheet rows (SUPERSEDED by R15-1 · sheet is terminal).
         · QuickProfileSheet itself · the sheet's primary subject (R15-1
             friend variant is terminal).
         · QuickProfileSheet mutual-friend avatars (RESOLVED to a stacked
             QuickProfileSheet per R16-3 · NOT a deep push to Friend
             Profile).
         · InboundBroadcastCard row body · prior intent (DESIGN-BACKLOG
             line 281) is DEFERRED to a future round and remains stubbed.
       NEVER add a Friend Profile entry point from any sheet surface in
         this round. NEVER allow a deep link or push notification to
         route to Friend Profile for a non-friend target — the route
         handler must reject with NOT_FRIEND and surface an ErrorState.

R16-3. QUICKPROFILESHEET MUTUAL-FRIEND AVATAR TAP STACKS A SECOND
       QUICKPROFILESHEET (DEPTH-1 CAP). Tapping a mutual-friend avatar
       inside the mutual-friends row of any QuickProfileSheet opens a
       new QuickProfileSheet for the tapped user, layered above the
       current sheet. The new sheet's variant (friend vs. non-friend)
       resolves per R15-2 at tap time.
       Stacking behavior:
         · The previously open sheet remains mounted underneath; its
           backdrop opacity is NOT re-applied (the existing 30% backdrop
           from R12-5 covers both sheets).
         · The new sheet animates in using the same entrance motion as a
           first-level QuickProfileSheet (durations.sheetUp · spring per
           R12-5).
         · Dismissal of the top sheet (swipe-down per R13-1 OR tap-X)
           reveals the underlying sheet in place. The underlying sheet's
           scroll position, expanded-bio state, and optimistic
           friend-request state are preserved.
         · Dismissal of the underlying sheet (only reachable after the
           top sheet is closed) returns to the host surface (Search
           overlay or AttendeesSheet).
       Depth cap: exactly ONE level of nesting is permitted. Inside the
         depth-1 sheet, mutual-friend avatars render but tap is a no-op
         (no haptic, no navigation, accessibilityState.disabled = true).
         This prevents runaway nesting and memory bloat.
       Haptic: depth-0 → depth-1 transition fires `light` (same as
         QuickProfileSheet open). No haptic on depth-1 mutual taps.
       NEVER stack a third sheet. NEVER dismiss the underlying sheet
         while the top sheet is open. NEVER carry optimistic state across
         sheet levels — each sheet owns its own friendRequest local state.

R16-4. SELF-VIEW GUARD ON MUTUAL-FRIEND AVATARS. If the userId on any
       mutual-friend avatar (in any QuickProfileSheet at any depth)
       matches currentUser.id, the avatar is rendered NON-INTERACTIVE:
         · Pressable is replaced by a plain View (or
           accessibilityState.disabled = true).
         · No haptic, no navigation, no visual hover/press state.
         · The avatar still renders with its status ring (per R15-2) so
           the user can verify their own appearance in someone else's
           mutual list — this is informational only.
       The same guard applies to Friend Profile's MUTUAL EVENTS section
         row taps (which are read-only per R16-11 regardless, but the
         guard formalises the rule for any future drill-through).
       NEVER push FriendProfile with friendId === currentUser.id. The
         route handler must reject with NOT_FRIEND.

R16-5. FRIEND PROFILE ACTION ROW IS TWO EQUAL PRIMARY PILLS. The bottom
       of FriendProfileScreen renders a single row of two equal-flex
       PillBtns side-by-side:
         · "Make Plans" · variant=primary · size=md · leading icon
             calendar-plus (16px) · onPress → opens CreateEventStack
             with `prefilledInviteeIds: [friendId]` · medium haptic.
         · "DM" · variant=primary · size=md · leading icon message
             (16px) · onPress → DM stub per R16-9 · light haptic.
       8px horizontal gap between pills. No destructive action sits in
         the action row — Remove friend, Block, and Report all live in
         the header overflow per R16-6.
       Make Plans and DM share equal visual weight (both primary). This
         is intentional pending R16-9; revisit if DM has not been
         promoted to a real system within the next major round.
       The action row sits below all other sections (R16-1 step 6) and
         is the LAST element before bottom safe-area inset. It is NOT
         pinned; it scrolls with the body content.
       NEVER place Remove friend, Block, or Report in the action row in
         this round — all three live in the header overflow (R16-6).
         NEVER show a single full-width primary pill in the action row
         (would reintroduce the "Plan together" only layout).

R16-6. FRIEND PROFILE HEADER OVERFLOW HOSTS REMOVE FRIEND + BLOCK +
       REPORT. The FlowHeader trailing slot on FriendProfileScreen
       renders a ⋯ overflow icon (Ionicons "ellipsis-horizontal" or
       equivalent Tabler icon, 20px ink2). Tap opens a sheet-style
       overflow menu (re-use RowOverflowMenu pattern from
       src/components/social/) with exactly three rows, in this order:
         (a) "Remove friend" · destructive row · onPress →
             TwoTapDestructive confirm → useRemoveFriend mutation per
             R16-7.
         (b) "Block {firstName}" · destructive row · onPress →
             TwoTapDestructive confirm → useBlockUser mutation per
             R16-8.
         (c) "Report {firstName}" · neutral row · onPress → Report stub
             per R16-9.
       Order rationale: rows are ordered by escalation gradient.
         Remove friend is the lightest and most common action
         (reversible — friend request can be sent again). Block is the
         heavier consequence (severs the relationship and hides content
         system-wide). Report is a separate concern (about behavior,
         not relationship status) and sits last.
       The overflow opens with a `light` haptic. Each row tap fires its
         own haptic per the action's spec (heavy on Remove and Block
         arms via TwoTapDestructive; success on Report confirm per
         R16-9). Dismiss tap on backdrop closes the menu without action.
       Both destructive rows (Remove friend, Block) use the SAME visual
         treatment — destructive row anatomy (red ink). The escalation
         is communicated by order, not by color weight.
       NEVER expose Remove friend or Block as a one-tap action — both
         must pass through TwoTapDestructive (Hard Rule 7). NEVER add a
         fourth row in this round (no Mute, no Hide events, no Share
         profile).

R16-7. REMOVE FRIEND IS A REAL MUTATION WITH SILENT RETURN. The Remove
       friend row in the header overflow (R16-6 row (a)) opens a
       TwoTapDestructive confirm, then on confirm invokes
       `useRemoveFriend(friendId)`. On success:
         · Cache invalidates `friends.all()` and `friends.profile(id)`.
         · The overflow menu closes if still open. `navigation.popToTop()`
           returns the user to FriendsList.
         · NO toast renders. NO confirmation banner renders. The removed
           friend simply disappears from the list — the absence is the
           confirmation. This matches the rest of SyncUp's destructive-
           action language (TwoTapDestructive is itself the confirmation
           gesture; a post-success toast would be redundant).
       On error:
         · Show ErrorToast `kind='generic'`. Stay on FriendProfileScreen
           (overflow remains closed; the user can re-open it to retry).
         · The TwoTapDestructive in the overflow row re-arms on next open.
       Haptics: TwoTapDestructive handles arm (heavy) and commit (success)
         internally — no additional firing from the screen.
       NEVER show a "Removed {name}" toast in this round. NEVER auto-
         navigate before the mutation resolves (no optimistic pop).
         NEVER allow Remove friend to be a one-tap action — must always
         pass through TwoTapDestructive (Hard Rule 7).

R16-8. BLOCK USER IS A REAL MUTATION WITH SILENT RETURN. The Block row
       in the overflow (R16-6) opens TwoTapDestructive confirm, then on
       confirm invokes `useBlockUser(friendId)`. On success:
         · Cache invalidates `friends.all()`, `friends.profile(id)`,
           and any feed query that surfaces user-authored content
           (events, broadcasts). A new query key family
           `blockedUsers.list()` is introduced.
         · `navigation.popToTop()` returns to FriendsList.
         · NO toast. Same silent-return language as R16-7.
       On error: same as R16-7 (ErrorToast generic, stay on screen).
       Block implicitly removes the friendship — the backend treats Block
         as a superset of Remove. The UI does NOT call useRemoveFriend
         separately.
       Haptics: TwoTapDestructive handles arm + commit internally.
       NEVER expose Block without TwoTapDestructive. NEVER show an
         explicit "blocked" affordance on Friend Profile (the screen is
         no longer reachable for blocked users — they're filtered out of
         friends list).

R16-9. DM AND REPORT ARE STUBS UNTIL THEIR BACKING SYSTEMS LAND. In
       this round, DM and Report are intentionally shipped as stubs:
         · DM tap → fires `light` haptic + renders a transient toast:
             "Direct messaging is coming soon."
             Toast variant: standard info · 2200ms duration · no Undo.
         · Report tap → opens TwoTapDestructive confirm (single confirm
             step, "Tap again to report"). On confirm: fires `success`
             haptic + renders a transient toast:
             "Thanks — we'll review this report."
             No mutation is called. No server roundtrip. The user-facing
             affordance exists; the backend wiring does not.
       Both stubs are documented in DESIGN-BACKLOG.txt with their
         follow-up systems (DM = full messaging surface; Report = real
         moderation pipeline). When either real system lands, this rule
         is superseded for that affordance.
       The visual treatment of DM in the action row (R16-5) is FULL
         PRIMARY pill weight despite being a stub. This is a known
         tradeoff: equal visual weight may set user expectations the
         feature cannot meet in this round. Flagged for revisit if DM
         is not promoted to a real system in the next major round.
       NEVER ship a "Coming soon" toast for more than one major round
         without either promoting the feature or removing the button.
         NEVER call any DM or Report endpoint in this round — both are
         pure client-side stubs.

R16-10. FRIENDTYPE CHIPS ON FRIEND PROFILE ARE DISPLAY-ONLY. The FRIEND
       TYPES section on FriendProfileScreen renders zero-or-more chips,
       one per FriendType containing this friend. Chip anatomy is
       identical to existing screen (caption text, accentSoft fill,
       hair border, radius pill).
       Tap on a chip is a NO-OP — no haptic, no navigation. Editing
         FriendType membership lives entirely in FriendTypesManager
         (route: FriendTypesManager). To change which types a friend
         belongs to, the user must navigate there.
       The "Friend type assigned (chip toggle here) → medium" haptic
         documented in SCREENS.md is RESCINDED in this round — that
         haptic only applies inside FriendTypesManager.
       NEVER allow inline toggle of FriendType membership on Friend
         Profile in this round.

R16-11. MUTUAL EVENTS ON FRIEND PROFILE ARE READ-ONLY. The MUTUAL EVENTS
       section on FriendProfileScreen renders past + upcoming events
       where both currentUser and friendId are invitees.
       Each event renders as a stacked row with:
         · Event title · bodyMed ink · 1-line clamp.
         · Event ISO timestamp · caption ink2.
       Tap on an event row is a NO-OP — no haptic, no navigation. The
         list is informational only.
       Empty state renders EmptyMutualEvents per existing screen.
       Future round may add tap-to-EventDetail; until that is specced
         explicitly, the list remains read-only to keep Friend Profile
         minimal and the navigation graph predictable.
       NEVER push EventDetail from Friend Profile in this round.

## HARD RULES — Round 15 additions (AttendeesSheet row-tap + scale · Onboarding stack)
================================================================
R15-1. ATTENDEEROW TAP OPENS QUICKPROFILESHEET (TERMINAL). Tapping any
       AttendeeRow body in AttendeesSheet opens QuickProfileSheet
       stacked above the sheet. The friend/non-friend variant is
       resolved at tap time (see R15-2). QuickProfileSheet's scope
       (R12-5) extends from "Search overlay only" to include

R13-1. SHEET DRAG GESTURE IS VELOCITY-FIRST, DISPLACEMENT FALLBACK.
       The NotifSheet drag gesture (on PullDownAffordance or drag
       handle) resolves to a target detent in two phases:

       PHASE A — ENTRY (closed → first detent):
         Triggered by downward drag on PullDownAffordance.
         Sheet follows touch 1:1 from the first pixel of movement.
         Velocity ≥ 0.5px/ms (downward) at release → full directly.
         Displacement < 60px at release → spring back to closed.
         Displacement 60–159px → snap to peek (light haptic).
         Displacement ≥ 160px → snap to full (light haptic).

       PHASE B — INTER-DETENT (peek ↔ full ↔ closed):
         Triggered by drag on the sheet grab handle area (44pt hit).
         Sheet follows touch 1:1; clamps at translateY(12%) (full
         cap) and snaps closed at bottom of screen.
         Velocity-first rule: if |velocity| ≥ 0.7px/ms at release,
           snap-direction determined by velocity sign regardless of
           displacement:
             Downward velocity ≥ 0.7px/ms from full  → peek.
             Downward velocity ≥ 1.2px/ms from full  → close (skip peek).
             Downward velocity ≥ 0.7px/ms from peek  → close.
             Upward velocity   ≥ 0.7px/ms from peek  → full.
             Upward velocity   ≥ 0.7px/ms from full  → no-op
               (already at top; rubber-band 12px then spring back).
         Displacement fallback (|velocity| < 0.7px/ms):
             From peek: down >40px → close; up >40px → full.
             From full: down >80px → peek; down >200px → close.
             Inside thresholds → spring back to current detent.

       SNAP ANIMATION: 320ms spring cubic-bezier(0.34,1.56,0.64,1).
         Backdrop crossfades 280ms easeStd in lockstep with snap.
         Haptic fires at snap moment (not at pointer-up).

       REDUCED-MOTION FALLBACK FOR GESTURE PATH (A-11, A-21):
         Live tracking: unchanged (1:1 follow is not an animation).
         Snap animation: 200ms easeOut translate (no spring, no
           overshoot). Backdrop: instant switch at release.
         NEVER suppress gesture tracking itself — only the snap anim.

       NEVER require a tap shortcut as the only path between detents
         in production — the gesture is the primary affordance.

R13-2. NOTIFSHEET OFFLINE STATE IS SHOWN INLINE, ABOVE THE FEED.

      in-app card title and body exactly. No "New activity from" /
      "SyncUp:" / "You have a new" prefix.
        ✓ "Ana is free tonight"
        ✗ "New activity from Ana"

        ✓ "Maya wants to be friends"

      Group Detail body scrolls. Does NOT collapse, fade, shrink, or

Hard rules:

  H-2. NEVER stack haptics within 80ms (debounce in useHaptic).
  H-3. NEVER haptic on mount or auto-fired toasts not triggered by user.
  H-4. Two-tap destructive: heavy on arm, success on commit.
  H-5. Toasts fire haptic WITH the toast, not on dismiss.
