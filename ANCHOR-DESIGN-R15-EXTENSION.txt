## HARD RULES — Round 15 additions (AttendeesSheet row-tap + scale · Onboarding stack)
================================================================
R15-1. ATTENDEEROW TAP OPENS QUICKPROFILESHEET (TERMINAL). Tapping any
       AttendeeRow body in AttendeesSheet opens QuickProfileSheet
       stacked above the sheet. The friend/non-friend variant is
       resolved at tap time (see R15-2). QuickProfileSheet's scope
       (R12-5) extends from "Search overlay only" to include
       AttendeesSheet as a valid host surface. The sheet is TERMINAL
       in this round — no "See full profile" affordance, no drill-
       through to a deeper surface. Friend Profile routing from
       AttendeesSheet is deferred and tracked in DESIGN-BACKLOG.txt
       alongside the existing Friend Profile entry. SUPERSEDES the
       "Tap row → Friend Profile" clause in R10-7. The viewer's own
       row renders normally (with their RSVPBadge visible to others)
       but tap is a no-op: no haptic, no navigation.
       NEVER navigate away from AttendeesSheet on row tap in this round.
       NEVER stack QuickProfileSheet over a row owned by the viewer.

R15-2. QUICKPROFILESHEET HAS TWO VARIANTS RESOLVED BY RELATIONSHIP.
       Variant resolution: friend = the target appears in any of
       currentUser's Friend Type assignments (R7-3). Non-friend =
       everyone else (including the host of an event the viewer is
       attending, if they aren't yet friends).
         Non-friend variant: identical to R12-5 / R12-6 (unchanged).
         Friend variant differs ONLY in three ways:
           (a) RingAvatar 64px shows the status ring (viewer has
               permission per friendship · per existing token rules).
           (b) The CTA block (R12-5 · "Add friend" / "Requested" /
               Accept+Decline) is REPLACED by a single non-interactive
               Friend Type chip at the same vertical position:
               10/600 mono · accentSoft fill · accent text · radius
               999 · 6×10 padding · centered. Label = the Friend Type
               currentUser has assigned to the target (e.g.,
               "CLOSE FRIENDS").
           (c) If the target is untyped (no Friend Type assigned), the
               chip is OMITTED entirely. No placeholder. No "UNTYPED"
               label. No empty pill outline. The CTA slot is just blank.
         Body content (mutual friends · bio · Hosted/Attended stats)
         is identical to the non-friend variant.
       Friend variant is also TERMINAL per R15-1 — no Message, no
       View Shared Events, no Remove Friend in this surface.
       NEVER show Add/Requested/Accept/Decline controls in the friend
         variant. NEVER show the status ring on the non-friend variant.

R15-3. ATTENDEESSHEET LIST IS GROUPED BY RSVP WITH HOSTS PINNED ON TOP.
       The visible attendee list is a single scrollable column with
       section overlines in this fixed order:
         HOSTS · GOING · MAYBE · NOT GOING · NO RESPONSE
       Section overlines: mono 10/600 ink3 · letter-spacing 1.6 ·
         12px top padding · 6px bottom padding · left-aligned · 14px
         horizontal margin.
       RSVP section overlines include "({N})" count, e.g., "GOING (32)".
       HOSTS section overline does NOT show a count.
       HOSTS section order: the event host's row first, then co-hosts
         alphabetical by name (case-insensitive). The host row carries
         a "HOST" chip in the same position as the existing CO-HOST
         chip (R11-3) — anatomy: 10/600 mono · accentSoft fill ·
         accent text · radius 999 · 6×10 padding · right-aligned ·
         6px gap from RSVPBadge.
       Each host and co-host ALSO appears in their RSVP section below.
         The row appears twice in the list. The RSVPBadge and the
         HOST/CO-HOST chip travel with the row in both appearances —
         row anatomy is identical regardless of which section it sits in.
         Tap behavior (R15-1) is identical in both appearances.
       RSVP section counts include hosts (e.g., if the host RSVPed
         Going, they count toward "GOING (N)").
       Empty RSVP sections are OMITTED — never render an overline with
         "(0)" below it.
       The HOSTS section is OMITTED if no host/co-host matches the
         active chip filter (R10-2 / R10-3).
       This grouped structure applies to all three role views
         (HOST · CO-HOST · INVITEE per R11-7) — the only difference
         between views is which chip filter rows render (R10-3 /
         R11-7) and which row-level affordances render (R11-7).
       NEVER hide hosts from the RSVP sections — the double appearance
         is intentional. NEVER merge HOSTS into GOING.

R15-4. ATTENDEESSHEET SEARCH IS REVEALED FROM A HEADER MAGNIFIER.
       The search affordance is a magnifying-glass icon-btn
       (20px ink3 · 44pt hit) in the sheet header, immediately left
       of the Close X (HOST/CO-HOST views: left of the "Invite more"
       pill, which sits left of Close X).
       Tap behavior:
         Light haptic · sheet header's title row collapses upward
         200ms easeStd · a SearchInputBar slides into its place:
           Full-width pill · radius 12 · bgSunken · hair border ·
           auto-focus · placeholder "Search attendees…" ·
           clear × icon-btn inside right edge (only renders when
           text present · light haptic on tap).
         The magnifier becomes a "Cancel" text link (13/500 accent)
           that exits search mode and restores the header.
         "Invite more" pill (HOST/CO-HOST) is hidden during search.
       While the search input has a non-empty query:
         Section overlines (HOSTS · GOING · MAYBE · NOT GOING ·
           NO RESPONSE) are HIDDEN entirely.
         All matching rows render in a single flat alphabetical list,
           sorted by name (case-insensitive).
         Matching is case-insensitive against name AND @handle.
         Hosts and co-hosts retain their HOST/CO-HOST chip but lose
           pinned position — sorting is strictly alphabetical.
         The viewer's own row, if matched, is included in the flat
           list (still non-interactive per R15-1).
       Clearing the query (tap × or empty input) restores the grouped
         structure with HOSTS pinned (R15-3).
       Exiting search mode (tap "Cancel") clears the query and
         restores the header title row.
       Search input filters compose with chip filters (R10-2 · R10-3):
         the search input filters within whatever chip filter is active.
       Search is purely local against the loaded list — NEVER fires
         a network request (reaffirms R10-6).
       NEVER render section overlines during an active search.
       NEVER auto-clear the search input when scrolling.

R15-5. ATTENDEESSHEET FILTER CHIPS ARE STICKY ON SCROLL. The chip
       filter bar — 1 row for INVITEE/CO-HOST views (R11-7), 2 rows
       for HOST view (R10-3) — is sticky to the top of the sheet body
       as the list scrolls. The sheet header (title row OR search
       input per R15-4) remains pinned above the chip bar.
       Chips do not transform, fade, or compress on scroll — they
         remain at full opacity and full height at every scroll
         position.
       During an active search (R15-4): chips remain sticky AND remain
         interactive. Search and chip filters compose.
       Reduced-motion: behavior unchanged (sticky positioning is not
         an animation).
       NEVER scroll the chip filter bar off-screen.
       NEVER collapse the host's two-row chip stack into one row at
         any scroll position.

R15-6. ATTENDEESSHEET OFFLINE STATE MIRRORS NOTIFSHEET. When the
       device loses network while AttendeesSheet is open, or it is
       opened while offline:
         OfflineBar (R5-7) renders immediately below the sheet header
           and above the chip filter bar.
         "SYNCED {T} AGO" sub-line renders below OfflineBar in mono
           10/600 ink3. Omit if last sync time is unavailable.
         The attendee list remains visible (stale data > blank state).
       Network-dependent actions are disabled (opacity 0.4 · pointer-
       events: none):
         "Invite more" pill (R10-5 · R11-1a · HOST + CO-HOST views).
         The host's ⋯ menu items (R11-2) — entire menu becomes
           non-tappable; tapping the ⋯ icon is a no-op with light
           haptic suppressed.
         Any already-armed remove (R10-4 / R11-4) auto-cancels back
           to default state if network drops mid-arm; no commit fires.
       Local actions remain active: scroll · chip filter taps ·
         search reveal/clear · row tap → QuickProfileSheet (profile
         data fetch may fail and surface its own loading/error state
         per R12-5).
       On reconnect: OfflineBar slides out (su-slide-up 240ms easeStd)
         · stale sub-line hides · list refreshes via React Query
         invalidation · any newly added attendees animate in via
         StaggerList.
       NEVER show two OfflineBars simultaneously (E-1 unchanged).
       NEVER blank the list while offline.

R15-7. ONBOARDING FLOW ORDER IS LOCKED. The complete onboarding
       sequence, in order, with no inserted screens or sub-routes:
         Welcome →
         Sign-Up Step 1 · Credential →
         Sign-Up Step 2 · OTP Verify →
         Sign-Up Step 3 · Name + Handle →
         Sign-Up Step 4 · Password →
         Sign-Up Step 5 · Profile Photo (skippable · R9-7) →
         Sign-Up Step 6 · Availability (skippable · R9-7 · inline
           editor) →
         Native push notification permission modal (R15-9) →
         Friend-Find (R15-10 · skippable) →
         "You're In" Completion (R15-12) →
         Tap "Go" → Home tab root (default) OR Event Detail (if
           InviteContext is set · per R9-8).
       Sign-In and Forgot-Password are SEPARATE flows reached from
         the Welcome screen. They do NOT enter the 6-step sign-up
         progression and have no 6-dot indicator.
       The 6-dot indicator (R9-2) applies ONLY to Sign-Up Steps 1–6.
         Friend-Find, "You're In", Sign-In, and Forgot-Password do
         NOT show progress dots.
       NEVER add a "Welcome to SyncUp" celebration screen between
         sign-up steps and friend-find — "You're In" is the ONLY
         celebratory beat.
       NEVER reorder these screens. NEVER insert wrapper screens
         around the push permission modal. NEVER move Friend-Find
         after "You're In".

R15-8. ONBOARDING SIGN-UP CREDENTIAL FIELD IS A SINGLE FORMAT-
       AGNOSTIC INPUT. Sign-Up Step 1 uses ONE input that accepts
       phone OR email — no tabs, no segmented control, no format
       picker. Format detection mirrors R9-9 (Sign-In identifier):
         Keyboard type adapts based on first character or accumulated
         pattern: digit → numeric · letter or @ → default · letters
         containing '.' → email.
         Validation: input must match phone regex OR email regex (one
         or the other) before "Continue" enables (R9-4).
         Real-time visual: input border lights to accent the moment
         a valid format is matched; reverts to hair when not matched.
         Inline error appears ONLY after a failed submit attempt
         (R9-4). Error copy: "Enter a phone number or email."
       NEVER render a phone/email toggle on Step 1.
       NEVER reject a format the user hasn't finished typing.
       NEVER block submission until both formats validate (one is
         enough).

R15-9. PUSH NOTIFICATION PERMISSION USES THE NATIVE OS PROMPT WITH
       NO WRAPPER. The push permission ask is the native iOS
       Notifications system prompt, fired automatically immediately
       after Sign-Up Step 6 completes OR is skipped — and before
       Friend-Find renders. No SyncUp screen wraps the prompt; no
       "Why we ask for notifications" pre-screen; no post-prompt
       confirmation screen.
       Outcome handling:
         Granted: silently advance to Friend-Find. No haptic, no toast.
         Denied: silently advance to Friend-Find. No haptic, no toast,
           no in-app warning.
       Re-prompt logic: SyncUp does NOT re-prompt during onboarding
         regardless of outcome. iOS only allows the system prompt once;
         further changes happen in Settings. A contextual deep-link to
         Settings MAY appear later if the user interacts with a
         notification preference (e.g., per-card mute) — that flow is
         outside R15 scope and tracked separately.
       NEVER wrap the system prompt in a SyncUp screen.
       NEVER fire the prompt before Step 6 completes — the user has
         not yet seen enough value to grant.
       NEVER fire it twice in one onboarding session.

R15-10. FRIEND-FIND IS A THREE-BRANCH STEP. Friend-Find renders
        after the native push permission modal resolves and before
        "You're In". A single Decision screen branches into one of
        three paths:
          "Find friends" tap → triggers native iOS Contacts permission
            prompt:
              System grants  → push Matches List.
              System denies  → push "No worries" prompt.
          "Not now" tap     → push "No worries" prompt directly
            (does NOT trigger the system permission prompt).
          "Skip" tap        → push "No worries" prompt directly
            (does NOT trigger the system permission prompt).
        The "No worries" prompt is the unified denial/skip
          destination — all three "no" paths land in the same place
          with the same copy and the same exit ("Got it" → "You're In").
        The Matches List shows contacts who already have SyncUp
          accounts. Match per row is a tap-to-add interaction (same
          pattern as Search PEOPLE row · R8-7). Row body tap is a
          no-op during onboarding — QuickProfileSheet is NOT opened
          on this screen to keep the surface focused. The "Continue"
          pill is always enabled (adding zero is permitted).
        Zero-matches state (permission granted, no contacts matched):
          Render an inline EmptyState (users glyph + "No one yet." +
          "Friends will show up here once they join SyncUp.") with
          "Continue" pill below.
        NEVER show QuickProfileSheet during onboarding Friend-Find.
        NEVER block "Continue" on selection count.
        NEVER upload the user's contact book to the server — matching
          is performed by hashing phone/email locally and querying a
          server-side lookup endpoint (engineering pattern; preserves
          the "Nothing is uploaded." copy).

R15-11. CONTACTS-DENIED STATE IN FRIENDS TAB DEEP-LINKS TO SETTINGS.
        When the user has denied iOS Contacts permission (either
        during onboarding via "Find friends" → system Deny, or later
        via Settings), the Friends tab renders a small persistent
        affordance below the search bar:
          "Find friends faster — Turn on contacts"
          (12/500 ink3 · "Turn on contacts" in accent · left-aligned ·
           8px below the search input · 14px horizontal margin).
        Tap target is the "Turn on contacts" phrase only. Light
          haptic on tap. Tap fires Linking.openSettings() — user is
          taken directly to SyncUp's iOS Settings page where Contacts
          toggle lives. App backgrounds; no in-app modal renders.
        Render conditions:
          Status = denied:        affordance renders.
          Status = not_determined: affordance does NOT render. Friends
                                   tab empty state uses the standard
                                   "Find friends from contacts" CTA
                                   that re-enters the R15-10 Decision
                                   screen flow.
          Status = granted:       affordance does NOT render.
        NEVER deep-link to Settings without a prior failed permission
          attempt (the user hasn't been asked yet — ask first).
        NEVER render the affordance permanently — it disappears the
          moment the user grants permission.

R15-12. "YOU'RE IN" COMPLETION IS THE LAST PRE-APP BEAT. Anatomy:
        96px RingAvatar centered · 96px below safe-area top.
          If user uploaded a photo in Step 5: their photo · NO status
            ring (status not displayed here even if Step 6 completed —
            this is a celebratory moment, not a profile view).
          If user skipped photo: default initials avatar (accentSoft
            fill · accent text · 30/800 initials).
          A 24×24 lime-fill circle with white checkmark glyph docks
            on the avatar's bottom-right: offset 4px down · 4px right
            · 2px white border for separation against the avatar edge.
        Heading: "You're in, {FirstName}." stepTitle (24/800/-0.5) ·
          centered · 24px below avatar · {FirstName} = first word of
          the display name from Step 3, trimmed.
        Sub: "Now go plan something." (15/500 ink2 · centered · 8px
          below heading).
        CTA: "Go" accent PillBtn (full-width · pinned 32px above
          safe-area bottom).
        Tap "Go" routing (per R9-8 · this rule generalizes the routing
          decision):
          InviteContext set → push Event Detail for InviteContext.eventId
            and mount the main app shell.
          No InviteContext     → push Home tab root and mount the main
                                   app shell.
        Success haptic fires WITH the "Go" tap (one haptic only — not
          a sequence of light+success).
        Entrance: avatar + checkmark animate via `flow-fade-up`
          keyframe (320ms spring · token-defined). Reduced-motion
          fallback per A-11 / A-21: 200ms easeOut translate, no
          overshoot.
        This screen has NO back arrow and NO 6-dot indicator.
        NEVER add a "Maybe later" or "Explore" alternate CTA — "Go"
          is the single exit.
        NEVER auto-navigate from this screen — user must tap "Go".

R15-13. FIRST-RUN EMPTY STATES SHOW TUNED COPY UNTIL THE USER TAKES
        THEIR FIRST MEANINGFUL ACTION. A user is "first-run" when
        all three of these counters are zero:
          User.stats.hosted   === 0
          User.stats.friends  === 0
          User.stats.groups   === 0
        The transition out of first-run is one-way and global. The
        instant any counter becomes non-zero (creating an event ·
        accepting a friend · creating or joining a group), standard
        empty-state copy applies to every tab from that moment
        forward. There is no manual dismiss, no time-based expiry,
        no per-tab transition.

        First-run state applies to FOUR tab-level surfaces only:
          Home (Today / Week / Month views — single shared copy)
          Friends tab → Friends segment
          Friends tab → Groups segment
          Activity tab → NotifSheet body
        Each renders the existing empty-state component (EmptyHome,
        EmptyFriends, EmptyGroups, EmptyNotifications) unchanged in
        anatomy and CTAs — only the title and body strings differ
        from standard.

        First-run copy (LOCKED):
          Home          Title: "Plan your first event."
                        Body:  "Tap + to start."
                        CTA:   "Plan something" (existing).
          Friends seg.  Title: "Find your first friend."
                        Body:  "Search by @handle or share your QR
                                for them to scan."
                        CTAs:  "Add one" + "Share QR" (existing).
          Groups seg.   Title: "Start a group for your regulars."
                        Body:  "Easier than tagging the same people
                                every time."
                        CTA:   "Create one" (existing).
          Activity      Title: "Nothing yet."
                        Body:  "Add friends or plan an event to
                                start seeing activity."
                        CTA:   none (EmptyNotifications NO-CTA lock
                                unchanged).

        Invite-context users (R9-8) land on Event Detail, not Home,
        and therefore never see first-run Home as their first
        impression. When they navigate to Friends, Groups, or
        Activity tabs, first-run copy applies per the rules above.
        The InviteContext event itself does NOT count toward
        User.stats.hosted (the user is an attendee, not the host);
        thus invite-context users remain first-run until they take
        a creator action.

        Composition with other affordances:
          Friends-tab contacts-denied affordance (R15-11) sits above
            the first-run EmptyFriends. Both render simultaneously
            when the user is denied + first-run + has zero friends.
          AudiencePickerSheet zero-friend state (R13-4) is a sheet-
            level empty and is NOT first-run gated — it applies any
            time the friend list is empty, regardless of first-run.

        NEVER add a separate nudge card, coachmark, modal,
          checklist, or overlay for first-run guidance.
        NEVER modify the empty-state component anatomy — first-run
          differs in copy only.
        NEVER apply first-run copy to Broadcast (always populated
          with the 3 Free/Maybe/Busy cards) or Profile (always
          populated with the user's identity from onboarding).
        NEVER apply first-run copy to non-tab surfaces (Search
          overlay, AvailabilityEditor, AttendeesSheet, etc.).
        NEVER persist a "first-run dismissed" flag — the state
          ends only via the User.stats counter logic above, never
          by user dismissal.

## ONBOARDING SURFACE (LOCKED · Round 15)
================================================================
Stack: AuthStack — mounted by RootNavigator when currentUser is null
  (unauthenticated). The main app shell mounts ONLY after successful
  authentication AND completion of "You're In" (tap "Go").
Navigation: native-stack (no bottom tabs · R9-1).
Screen wrapper: every Sign-Up Step (1–6), Sign-In, Forgot-Password
  Entry, and Forgot-Password Confirm compose against the existing
  `AuthStep` component from R9 — it owns the back arrow row, 6-dot
  indicator (when step/totalSteps are passed), heading/sub stack,
  scrollable children area, and the bottom-pinned primary + secondary
  CTA row. R15 screen specs that follow describe the contents of the
  `children` slot and which AuthStep props each screen passes; they
  do NOT reinvent the layout. Welcome, Friend-Find (all 3 branches),
  and "You're In" do NOT use AuthStep — they are standalone screens
  because their layouts diverge from the step pattern.
Status bar: matches FLOW_TOKENS surface (light content over dark bg ·
  dark content over light bg).
Safe areas: respected top and bottom.
Background: bg token across all screens · no gradient · no artwork.
6-dot indicator (R9-2): rendered on Sign-Up Steps 1–6 ONLY.
Back arrow (R9-3): 44pt · 24px ink3 · top-left · preserves form state.
  Welcome has NO back arrow (R9-3 exception).
  Forgot-Password Confirm has NO back arrow (R9-10).
  "You're In" has NO back arrow (R15-12).

## OPEN QUESTIONS
================================================================
The following are deliberately NOT locked in R15 and need follow-up
before build. Each is flagged inline above; consolidated here for the
build team.

1. PASSWORD REQUIREMENTS — R15 surfaces "8+ chars AND one number or
   symbol" as the visible UX bar. Confirm Clerk's enforced minimum
   matches; if Clerk requires stricter (e.g., length 10, banned-
   password list), the visible requirement list and inline checkmark
   logic must be updated.

2. OTP RESEND COUNTDOWN — locked at 0:30 in R15. Confirm with
   engineering / SMS provider whether 30 seconds is the appropriate
   throttle window.

3. PHOTO CROPPING — R15 locks square 1:1 crop on both camera capture
   and library pick. Confirm whether iOS's built-in crop UI satisfies
   this or an in-app cropper is needed. Image upload size /
   compression pipeline is implementation detail (not specified here).

4. CONTACT-MATCHING MECHANISM — R15-10 specifies "hashing phone/email
   locally" so the privacy copy "Nothing is uploaded." is honest.
   Engineering must confirm the matching endpoint accepts hashes, not
   raw values.

5. APPLE SIGN-IN / GOOGLE OAUTH — explicitly NOT in R15 scope. If
   added later, Welcome and Sign-In need a separate "Or continue with"
   block above the legal microcopy. Apple's App Store rules require
   "Sign in with Apple" if any other third-party OAuth is offered.

6. TOS / PRIVACY LINK BEHAVIOR — opening the Welcome screen legal
   links uses an external WebView. Whether this is SFSafariViewController,
   custom WKWebView, or external Safari is engineering choice and not
   locked here.

7. RE-PROMPT FOR CONTACTS PERMISSION — R15-11 deep-links to Settings
   from Friends tab. Whether the app should also offer a contextual
   re-prompt elsewhere (e.g., AudiencePickerSheet zero-friend state ·
   R13-4) is NOT addressed here. Recommend a follow-up round.

8. HANDLE TERMINOLOGY — R15 uses "@handle" throughout. Whether the
   user-facing label should be "Username," "Handle," or "@handle" is
   not fully consistent across the spec. Lock once a single term is
   chosen app-wide.

9. INVITECONTEXT EXPIRY — R9-8 captures InviteContext before auth
   renders. R15 doesn't specify whether the context expires if
   onboarding is paused for hours/days (e.g., user backgrounds the
   app on Step 3 and returns the next morning). Recommend a product
   decision: 24h expiry? Never expire?

10. MATCHES LIST SORT ORDER — R15-10 surface spec defaults to
    alphabetical by name (case-insensitive) for predictability.
    Confirm before build. Alternative: friends-of-friends weighted
    first, then alphabetical.

11. RINGAVATAR DEFAULT INITIALS — the "default initials avatar"
    referenced in R15-12 ("You're In") assumes an existing default
    avatar component pattern. Confirm the component exists in
    src/components/foundation/ or src/components/social/; if not,
    a small spec follow-up is needed for initials extraction
    (first letter of first word? first letters of first two words?)
    and color assignment.

12. STANDARD (NON-FIRST-RUN) EmptyNotifications BODY LINE — R15-13
    locks the first-run body for EmptyNotifications ("Add friends or
    plan an event to start seeing activity.") but the existing
    EmptyNotifications spec defines the 13/500 ink2 body slot
    without quoting text for the standard (post-first-run) state.
    Lock the standard body line in a future round. Candidates that
    came up during R15-13 design: "Quiet for now." (short idle) or
    reusing the first-run body verbatim.
